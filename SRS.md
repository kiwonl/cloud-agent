## 1. 개요 (Overview)

### 1.1 목적
본 애플리케이션은 기존 AWS(Amazon Web Services) 기반 개발자들의 서비스 인프라 디지인을 분석하고, 사내 인프라 체크리스트 부준을 검증함과 동시에 Google Cloud Best Practice를 투영하여 **Google Cloud 환경에서 실행 가능한 IaC(Terraform) 코드를 자동 생성**해 주는 AI Agent 애플리케이션이다.

### 1.2 배경 및 문제 정의 (Background)
- **조직 배경**: 클라우드 운영팀은 개발자들의 AWS 인프라 아키텍처에 대해 체크리스트를 검증하고, 이를 만족하는 Google Cloud Best Practice를 가이드해야 할 의무가 있다.
- **Pain point (장애 요인)**: 내부 운영진 및 개발진 모두 AWS 환경에만 숙련되어 있어, 개발자들은 모든 설계 도안을 AWS 자원 기반으로 작성해 오며, 이로 인해 Google Cloud 최적화 제안 및 번역 검토에 큰 병목이 발생하고 있다.
- **해결 방안**: AWS 인프라 구성(이미지/설명)과 기본 체크리스트를 양도받고 이를 **자동 분석 및 사상(Mapping) 및 상호 질의**해 주는 대화형 AI 가설을 통해 지식 격차를 극복하고자 한다.

### 1.3 시스템 가동 범위
- 인프라 청사진(설명/이미지) 기반 AWS 자원 구조체 자동 분석
- 에이전트 추가 정보 수집을 위한 **사용자 되질문(Inquiry) 피드백 루프** 가동
- 내부 인프라 만족도 체크리스트 자동 대응 여부 검증
- AWS ➡️ GCP 하부 서비스 1:1 및 1:N 자동 사상 (Mapping)
- GCP 배포 가용 최적화 **Terraform 코드(HCL) 최종 가동 사출**

## 2. 시스템 아키텍처 (System Architecture)

### 2.1 분산 하이브리드 가설
- **Frontend Layer (프론트엔드)**: 
  - React + Vite 기반 대화형 및 대시보드 하이브리드 SPA
  - 비동기 에이전트 폴링 타격 (`fetch` API targeting `/run`)
- **Backend Layer (백엔드 에이전트)**:
  - Google ADK 프레임워크 기반 멀티 에이전트 체인
  - FastAPI 기반 고속 ASGI 게이트웨이 사출 (Port 8000)
  - LLM 모델 캔버스: LiteLlm 연동 Vertex AI 아키텍처 백본

## 3. 기능 요구사항 (Functional Requirements)

### 3.1 [Step 1] 인프라 명세 캡처 및 대화 제어 (`UploadPage`)
- **[요구사항 ID: F-UPL-01] 분석 프롬프트 트리거링**: 사용자가 기재한 AWS 아키텍처 설명서와 인프라 체크리스트를 에이전트 추론망으로 주입 및 가동(Analyze)하는 핸들 레이어를 구현한다.
- **[요구사항 ID: F-UPL-02] 정보 수집 되질문 트리거링 (Conversation Prompting)**: 에이전트가 최초 명세만으로 매핑 및 코드를 짤 수 없을 경우 **환류 루프를 가동하여 사용자에게 추가 질의를 출력 표시**하고, 사용자의 입력 답변을 수용하여 연속 분석을 고도화하는 피드백 루프를 보증한다.
- **[요구사항 ID: F-UPL-03] 실시간 트레이스 로더**: 인프라 분석이 진행되는 동안 화면에는 `Processing 로더 스피너`가 동적 출력되어야 하며, 에이전트 추론 반응 트레이스가 연속 인쇄된다.

### 3.2 [Step 2] 노드 사상 및 만족도 검증 (`MappingPage`)
- **[요구사항 ID: F-MAP-01] AWS/GCP 서비스 최적화 매핑**: 에이전트 계산본(`mappings` 객체) 데이터를 수신 후, Source(AWS) 자원과 Target(GCP) 동급 최적 서비스 노드를 1:1 대조형 그리드 뷰포트에 렌더링한다.
- **[요구사항 ID: F-MAP-02] 체크리스트 검증 타겟팅**: 사용자가 배인한 인프라 체크리스트 세부 항목(예: VPC 분리, 보안 규칙 등)이 매칭 아키텍처 내부에서 준수되고 있는지 여부를 판단 검증한다.
- **[요구사항 ID: F-MAP-03] HITL 인간 승인 배너 바인딩 (`awaitingApproval`)**: 인프라 분석 에이전트가 후속 IaC 생성으로 단항 전이하기 전 **"승인 대기 상태" 피드백**을 검출하면 화면 상단에 검증 배너 및 승인 확정 버튼을 렌더링하여 안전 가동 레이어를 확보한다.

### 3.3 [Step 3] 프리플라이트 적합성 리포트 (`ChecklistPage`)
- **[요구사항 ID: F-CHK-01] 안정성 진단 가시 패널**: 마이그레이션 실행 전 준수되어야 할 Google Cloud Best Practice 기반 검증 피드백 상태 마킹(`complete`, `warning`, `pending`)을 하이드레이션 피킹하여 렌더링한다.

### 3.4 [Step 4] 테라폼 소스 하이라이팅 출력 (`TerraformPage`)
- **[요구사항 ID: F-TF-01] GCP 배포 가능 단일 Terraform Manifest 사출**: 에이전트 최종 사출물에서 벡틱 마크다운 블록을 감지하여 내장 HCL 코드 하이라이트 컨테이너로 추출 및 가시 사출 투영한다.

## 4. 인터페이스 요구사항 (Interface Requirements)

### 4.1 클라우드 에이전트 인터페이스 (API HTTP REST)
- **엔드포인트**: `POST http://localhost:8000/run`
- **인플릭스 레이어 구조**:
  - `newMessage`: 에이전트 주입 메시지 (사용자 프롬프트, 질문에 대한 답변, 또는 HITL 확정 분기 텍스트 `"Yes, Approve."`)
- **출력 바인딩 레이어 명세**:
  - 에이전트 파서 레이어는 반드시 마크다운 코드 블록(` ```json `) 내부에 `mappings` 및 `checklist` 객체 어레이 배정을 보증함으로써 클라이언트 비동기 수신 리사이클 파싱이 성립되도록 한다.

## 5. 비기능 요구사항 (Non-Functional Requirements)

### 5.1 시각적 안정성 및 사용자 경험
- **반응 속도**: 백엔드 에이전트 통신 타임아웃 지표는 원활한 대화형 캔버스를 위해 임계치 최소 확보(비동기 await 레이어) 하도록 대리 탑재한다.
